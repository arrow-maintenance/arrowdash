* **Numba support for PyArrow:** Discussion on integrating a subset of PyArrow into the Numba JIT compiler to enable fast scientific calculations directly on Arrow format data. The community is weighing the maintenance burden against the benefits of direct access to Arrow C++ internals. [Thread](https://lists.apache.org/list?dev@arrow.apache.org:lte=2M:DiscussPython%20Numba%20support%20for%20PyArrow)

* **Arrow 24.0.0 release:** The community approved the release of Apache Arrow 24.0.0, which resolves over 250 issues across multiple language implementations. Minor verification issues on Almalinux 8 were identified but deemed non-blocking. [Thread](https://lists.apache.org/list?dev@arrow.apache.org:lte=2M:VOTE%20Release%20Apache%20Arrow%2024.0.0%20-%20RC0)

* **Donation of pyarrow-stubs:** A vote passed to donate the `pyarrow-stubs` codebase to the Apache Arrow project, bringing PEP 484 type annotations for PyArrow directly into the main repository. This move aims to ensure type definitions stay in sync with the core library and benefit from official community maintenance. [Thread](https://lists.apache.org/list?dev@arrow.apache.org:lte=2M:DiscussPYTHON%20Donate%20pyarrow-stubs%20to%20Apache%20Arrow)

* **Arrow ADBC 23 release:** The Arrow Database Connectivity (ADBC) subproject released version 23, including 41 bug fixes and improvements. This release updates the independent versions of various language drivers, including C#, Java, and Rust. [Thread](https://lists.apache.org/list?dev@arrow.apache.org:lte=2M:VOTE%20Release%20Apache%20Arrow%20ADBC%2023%20-%20RC1)

* **C Device Data Interface experimentality:** Developers discussed why the C Device Data Interface remains marked as "experimental" three years after its introduction. Key concerns include low adoption rates beyond CUDA and uncertainty regarding its design suitability for other hardware devices. [Thread](https://lists.apache.org/list?dev@arrow.apache.org:lte=2M:Experimentality%20of%20Arrow%20C%20Device%20Data%20Interface)

* **Flight SQL Prepared Statements update:** A proposal and vote are underway to add an `is_update` field to the Flight SQL protocol for prepared statements. This change allows servers to explicitly signal whether a statement should follow an execution or update path, improving client-side compatibility. [Thread](https://lists.apache.org/list?dev@arrow.apache.org:lte=2M:VOTEFlight%20SQL%20Add%20is_update%20field%20to%20ActionCreatePreparedStatementResult)

* **Dictionary-encoded extension types in IPC:** Research revealed that the current IPC schema cannot distinguish between a Dictionary with Extension values and an Extension with Dictionary storage. A proposal has been made to add metadata to the DictionaryEncoding table to resolve this ambiguity during deserialization. [Thread](https://lists.apache.org/list?dev@arrow.apache.org:lte=2M:IPC%20representation%20of%20dictionary-encoded%20extension%20types)

* **Canonical BigDecimal Extension Type:** Proposal for a new canonical extension type to support arbitrary-precision decimals with variable scale per row. This is intended to improve data integrity when moving data between Arrow-native engines and systems like PostgreSQL or Trino that support variable-scale numeric types. [Thread](https://lists.apache.org/list?dev@arrow.apache.org:lte=2M:DiscussFormat%20Proposal%20for%20a%20Canonical%20BigDecimal%20Extension%20Type)

* **ADBC Logo Proposal:** A proposal was introduced for a new logo for the ADBC subproject to differentiate it while keeping it within the Arrow brand family. Feedback has been positive, with suggestions to include trademark symbols as per Apache guidelines. [Thread](https://lists.apache.org/list?dev@arrow.apache.org:lte=2M:DISCUSS%20ADBC%20Logo%20Proposal)

* **LargeMap format addition:** Discussion on adding a `LargeMap` type to the Arrow specification to support 64-bit offsets. This would prevent casting issues in libraries like Polars that rely on Large* types for processing massive datasets. [Thread](https://lists.apache.org/list?dev@arrow.apache.org:lte=2M:DISCUSS%20LargeMap%20format%20addition)
