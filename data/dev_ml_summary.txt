* **Apache Arrow 24.0.0 released:** The community announced the release of version 24.0.0, which includes 259 resolved issues across the ecosystem. Discussion during the voting process highlighted timezone-related test failures in Gandiva on specific macOS versions, though these were not considered blockers for the release. [Thread](https://lists.apache.org/list?dev@arrow.apache.org:lte=2M:ANNOUNCE%20Apache%20Arrow%2024.0.0%20released)

* **Canonical BigDecimal extension type proposal:** A new proposal introduces a canonical extension type for arbitrary-precision decimals (BigDecimal) to support variable scale per row. This aim is to improve data integrity when moving data between Arrow and engines like PostgreSQL, Trino, and Oracle that utilize variable-scale numeric types. [Thread](https://lists.apache.org/list?dev@arrow.apache.org:lte=2M:DISCUSS%20Format%20Proposal%20for%20a%20Canonical%20BigDecimal%20Extension%20Type)

* **Partitioned bulk ingest API for ADBC:** A significant new API is being proposed for ADBC to support distributed bulk ingestion, allowing parallel writers (like Spark or Flink) to commit data atomically. This addresses consistency issues in RDBMS writes and provides a standardized way to interface with table formats like Iceberg and Delta Lake. [Thread](https://lists.apache.org/list?dev@arrow.apache.org:lte=2M:DISCUSS%20ADBC%20Partitioned%20Bulk%20Ingest%20API)

* **Enforcing UTF-8 for metadata:** Discussion is underway regarding whether the C++ implementation and the C Data Interface should strictly enforce UTF-8 validation for field and schema metadata. This would ensure better compatibility with the Flatbuffers specification and prevent hard-to-track errors in various language bindings. [Thread](https://lists.apache.org/list?dev@arrow.apache.org:lte=2M:Discuss%20Field%2Fschema%2Fcustom%20metadata%20restriction%20to%20UTF8%3F)

* **Donation of Arrow Erlang implementation:** The community voted to accept the donation of the Arrow Erlang library, which currently provides bindings to the Rust implementation. The process involved significant discussion regarding IP clearance and contributor license agreements for contributors with trivial changes. [Thread](https://lists.apache.org/list?dev@arrow.apache.org:lte=2M:VOTE%20Donate%20Apache%20Arrow%20Erlang)

* **AVX2-optimized Bloom Filter probes for C++:** Developers are discussing porting high-performance AVX2 Split Block Bloom Filter (SBBF) probes from the Go implementation to C++. Initial benchmarks using `xsimd` show significant performance improvements (up to 5-7x) for in-cache Parquet read operations. [Thread](https://lists.apache.org/list?dev@arrow.apache.org:lte=2M:DISCUSS%20AVX2%20SBBF%20probe%20for%20parquet%2Fbloom_filter.cc)

* **Rust implementation security policy:** A new security policy was proposed for the Arrow and Parquet Rust implementations to formalize how vulnerabilities are reported and handled. The policy aims to clarify the distinction between standard bugs and security-sensitive issues in light of increased security reporting for open-source projects. [Thread](https://lists.apache.org/list?dev@arrow.apache.org:lte=2M:RUST%20Discuss%20Security%20Policy)

* **IBM Db2 support via ADBC:** Engineers from IBM proposed adding Apache Arrow support for IBM Db2, starting with a new ADBC driver. The project will eventually explore support for Arrow Flight SQL to enable high-performance data transport for analytical workloads. [Thread](https://lists.apache.org/list?dev@arrow.apache.org:lte=2M:DISCUSS%20ADBC%20DB2%20Proposal%20Adding%20Apache%20Arrow%20Support%20for%20IBM%20Db2)

* **Flight SQL dialect-related SqlInfo codes:** A proposal suggests adding four new SqlInfo codes to the Flight SQL protocol to describe backend-specific SQL dialects. These codes would cover LIMIT/OFFSET grammar, NULL ordering, and literal forms, helping clients like the Spark ADBC data source emit correct SQL for pushdown operations. [Thread](https://lists.apache.org/list?dev@arrow.apache.org:lte=2M:DISCUSS%20FlightSQL%20Add%20dialect-related%20SqlInfo%20codes)

* **Flight SQL is_update field:** Discussion and voting are taking place to add an `is_update` boolean to the `ActionCreatePreparedStatementResult`. This allows servers to indicate if a query is a DDL/DML statement, helping drivers select the correct execution path without using client-side heuristics. [Thread](https://lists.apache.org/list?dev@arrow.apache.org:lte=2M:DISCUSS%20Flight%20SQL%20Add%20is_update%20field%20to%20ActionCreatePreparedStatementResult)
