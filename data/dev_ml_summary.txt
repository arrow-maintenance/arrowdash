* **Arrow 25.0.0 and Rust 59.1.0 releases:** The community approved major releases for the main project and the Rust implementation, resolving over 200 issues including Gandiva and PyArrow improvements. [Thread](https://lists.apache.org/list?dev@arrow.apache.org:lte=2M:Release%20Apache%20Arrow%2025.0.0%20-%20RC1) [Thread](https://lists.apache.org/list?dev@arrow.apache.org:lte=2M:Release%20Apache%20Arrow%20Rust%2059.1.0%20RC1)

* **Conbench v2 modernization:** Wes McKinney proposed a major update to the benchmarking dashboard using a Go backend and Svelte frontend to improve UI responsiveness and data scalability. [Thread](https://lists.apache.org/list?dev@arrow.apache.org:lte=2M:Status%20of%20Arrow%20conbench%20data%20%2B%20conbench%20OSS%20project)

* **Deprecating specialized Tensor IPC messages:** Discussion on deprecating the experimental Tensor and SparseTensor IPC messages in favor of the more widely supported canonical extension types. [Thread](https://lists.apache.org/list?dev@arrow.apache.org:lte=2M:Deprecate%20Tensor%20and%20SparseTensor%20messages%3F)

* **Canonical JSON schema representation:** Proposal for a human-readable JSON format for Arrow schemas to simplify integration with REST APIs and improve metadata ergonomics in ADBC. [Thread](https://lists.apache.org/list?dev@arrow.apache.org:lte=2M:JSON%20representation%20of%20Arrow%20schemas)

* **CI resource optimization:** The project successfully reduced its GitHub Actions consumption by roughly 30% after exceeding ASF Infrastructure limits on shared runner resources. [Thread](https://lists.apache.org/list?dev@arrow.apache.org:lte=2M:Consumption%20of%20ASF%20Shared%20GitHub-hosted%20Runner%20Resources)

* **DuckDB ADBC extension donation:** Columnar proposed donating a DuckDB ADBC client extension to the ASF, prompting discussion on consolidating similar community efforts under official governance. [Thread](https://lists.apache.org/list?dev@arrow.apache.org:lte=2M:ADBC%20Extension%20for%20DuckDB)

* **Variant type specification alignment:** Developers are coordinating a change to the Parquet format to allow the `value` field to be omitted in shredded Variant types, bringing it in line with the Arrow spec. [Thread](https://lists.apache.org/list?dev@arrow.apache.org:lte=2M:Allow%20VARIANT%20value%20field%20to%20be%20omitted)

* **C Data Interface safety:** Clarification that the C Data Interface must error out on UTF-8 field names containing embedded nul bytes to prevent security issues and crashes in C-style string handling. [Thread](https://lists.apache.org/list?dev@arrow.apache.org:lte=2M:Spec%20clarification%20on%20field%20names)
