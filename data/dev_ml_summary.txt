* **Arrow 25.0.0 release:** The community approved the release of version 25.0.0 covering 220 resolved issues; a 25.0.1 patch is already being discussed to improve `to_pylist` performance for Spark 4.2 users. [Thread](https://lists.apache.org/list?dev@arrow.apache.org:lte=2M:VOTE%20Release%20Apache%20Arrow%2025.0.0%20-%20RC1)

* **Conbench modernization (v2):** Wes McKinney proposed a major update to the benchmark tracking project, including a Go-based backend and a modern Svelte/TypeScript frontend to improve performance and maintainability. [Thread](https://lists.apache.org/list?dev@arrow.apache.org:lte=2M:Status%20of%20Arrow%20conbench%20data%20%2B%20conbench%20OSS%20project)

* **JSON representation of Arrow schemas:** Discussion has started on defining a canonical, human-readable JSON format for schemas to facilitate better ergonomics for REST APIs and ADBC metadata. [Thread](https://lists.apache.org/list?dev@arrow.apache.org:lte=2M:DISCUSS%20Format%20JSON%20representation%20of%20Arrow%20schemas)

* **Deprecating Tensor IPC messages:** Developers are considering the deprecation of experimental Tensor and SparseTensor messages in the IPC format in favor of existing FixedShapeTensor and VariableShapeTensor extension types. [Thread](https://lists.apache.org/list?dev@arrow.apache.org:lte=2M:Discuss%20IPC%20Deprecate%20Tensor%20and%20SparseTensor%20messages)

* **DuckDB ADBC extensions:** A new ADBC extension for DuckDB was introduced by Columnar, leading to a debate on the trade-offs between "hand-coded" and "AI-assisted" development and potential donation to the ASF. [Thread](https://lists.apache.org/list?dev@arrow.apache.org:lte=2M:ADBC%20Extension%20for%20DuckDB)

* **CI runner usage optimization:** Following a request from ASF Infrastructure, the project successfully reduced its GitHub Actions runner consumption by 27% to stay within foundation-wide resource limits. [Thread](https://lists.apache.org/list?dev@arrow.apache.org:lte=2M:Consumption%20of%20ASF%20Shared%20GitHub-hosted%20Runner%20Resources)

* **Variant type implementation and alignment:** Discussions continue on implementing Variant type support in Arrow C++, alongside a proposal to update the Parquet format to allow the `value` field to be optional when shredded. [Thread](https://lists.apache.org/list?dev@arrow.apache.org:lte=2M:Variant%20Type%20Support) | [Thread](https://lists.apache.org/list?dev@arrow.apache.org:lte=2M:DISCUSS%20Allow%20VARIANT%20value%20field%20to%20be%20omitted)

* **C Data Interface field name safety:** Fuzz testing revealed edge cases regarding nul bytes in UTF-8 field names, leading to a consensus that producers should error out rather than allowing potential crashes during import. [Thread](https://lists.apache.org/list?dev@arrow.apache.org:lte=2M:Spec%20clarification%20on%20field%20names)

* **Triage of aging issues:** Maintainers are calling for community help to reproduce and triage older bug reports in the issue tracker to reduce technical debt and clear the backlog. [Thread](https://lists.apache.org/list?dev@arrow.apache.org:lte=2M:Closing%20out%20old%20issues%20-%20extra%20help%20wanted)
