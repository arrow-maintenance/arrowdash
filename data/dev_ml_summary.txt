* **Arrow 25.0.0 release:** The community approved version 25.0.0, resolving 220 issues across various implementations. Discussions also addressed a potential 25.0.1 patch to improve PyArrow performance for Spark UDF serialization. [Thread](https://lists.apache.org/list?dev@arrow.apache.org:lte=2M:VOTE%20Release%20Apache%20Arrow%2025.0.0%20-%20RC1)

* **JSON representation of Arrow schemas:** Proposal for a canonical, human-readable JSON format for Arrow schemas to simplify Rest API contracts and ADBC metadata exchange. The discussion focused on balancing Flatbuffer compatibility with ease of use for extension types. [Thread](https://lists.apache.org/list?dev@arrow.apache.org:lte=2M:DISCUSSFormat%20JSON%20representation%20of%20Arrow%20schemas)

* **Conbench v2 benchmarking overhaul:** Wes McKinney demonstrated a rewrite of the benchmarking infrastructure using Go and modern web technologies to improve UI performance. The project aims to migrate historical results to DuckDB and Parquet for more scalable storage. [Thread](https://lists.apache.org/list?dev@arrow.apache.org:lte=2M:Status%20of%20Arrow%20conbench%20data%20%2B%20conbench%20OSS%20project)

* **Deprecating IPC Tensor and SparseTensor:** Developers discussed deprecating experimental Tensor messages in the IPC format in favor of canonical extension types. This move would reduce security maintenance baggage and consolidate around more widely used tensor representations. [Thread](https://lists.apache.org/list?dev@arrow.apache.org:lte=2M:DISCUSSIPC%20Deprecate%20Tensor%20and%20SparseTensor%20messages%3F)

* **ADBC extension for DuckDB:** A hand-coded ADBC extension for DuckDB was proposed for donation to the project to facilitate connections to systems like Snowflake and BigQuery. The proposal initiated a debate regarding collaboration with existing community-led DuckDB extensions. [Thread](https://lists.apache.org/list?dev@arrow.apache.org:lte=2M:ADBC%20Extension%20for%20DuckDB)

* **Variant type support in C++:** Progress was made on mirroring the Rust implementation of the Variant type within the C++ library to ensure cross-language parity. This includes support for both shredded and unshredded data types within Parquet-cpp. [Thread](https://lists.apache.org/list?dev@arrow.apache.org:lte=2M:Variant%20Type%20Support)

* **Parquet Variant field omission:** Discussion on modifying the Parquet format specification to allow the Variant `value` field to be omitted when data is shredded. This would align Parquet more closely with Arrow's canonical extension type requirements. [Thread](https://lists.apache.org/list?dev@arrow.apache.org:lte=2M:DISCUSS%20Allow%20VARIANT%20value%20field%20to%20be%20omitted)

* **CI resource efficiency:** The project successfully reduced its GitHub Actions runner usage by nearly 30% to meet new ASF infrastructure policies. Maintainers emphasized continued vigilance to ensure shared resources remain available for all Apache projects. [Thread](https://lists.apache.org/list?dev@arrow.apache.org:lte=2M:Consumption%20of%20ASF%20Shared%20GitHub-hosted%20Runner%20Resources)

* **Community-wide bug squashing:** A new initiative was launched to triage stale issues and reproduce old bugs against recent Arrow versions. Contributors and maintainers are using a shared spreadsheet to coordinate the cleanup of the issue tracker. [Thread](https://lists.apache.org/list?dev@arrow.apache.org:lte=2M:Closing%20out%20old%20issues%20-%20extra%20help%20wanted)

* **Rust Object Store releases:** The Rust implementation released versions 0.14.0 and 0.14.1 in quick succession to address integration gaps and improve storage service compatibility. [Thread](https://lists.apache.org/list?dev@arrow.apache.org:lte=2M:VOTERUST%20Release%20Apache%20Arrow%20Rust%20Object%20Store%200.14.1%20RC1)
