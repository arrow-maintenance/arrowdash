* **Arrow 25.0.0 release:** The community approved the release of version 25.0.0, including a discussion on balancing release timing against a high-impact PyArrow performance fix for Spark 4.2 users. [Thread](https://lists.apache.org/list?dev@arrow.apache.org:lte=2M:VOTE%20Release%20Apache%20Arrow%2025.0.0%20-%20RC1)

* **Conbench v2 infrastructure overhaul:** Wes McKinney proposed a modernization of the project's benchmarking infrastructure using a Go-based backend and Svelte frontend to improve UI ergonomics and result storage performance. [Thread](https://lists.apache.org/list?dev@arrow.apache.org:lte=2M:Status%20of%20Arrow%20conbench%20data%20%2B%20conbench%20OSS%20project)

* **Proposed deprecation of Tensor IPC messages:** Discussion on deprecating experimental Tensor and SparseTensor messages in the IPC format, suggesting that users move to canonical extension types like FixedShapeTensor instead. [Thread](https://lists.apache.org/list?dev@arrow.apache.org:lte=2M:DiscussIPC%20Deprecate%20Tensor%20and%20SparseTensor%20messages%3F)

* **Canonical JSON representation for schemas:** Proposal to establish a standard human-readable JSON format for Arrow schemas to improve interoperability for REST APIs and metadata exchange in ADBC. [Thread](https://lists.apache.org/list?dev@arrow.apache.org:lte=2M:DISCUSSFormat%20JSON%20representation%20of%20Arrow%20schemas)

* **DuckDB ADBC extension donation:** Columnar-tech expressed interest in donating a new DuckDB ADBC extension to the ASF, leading to a debate regarding governance and the technical merits of "hand-coded" vs. "AI-assisted" development. [Thread](https://lists.apache.org/list?dev@arrow.apache.org:lte=2M:ADBC%20Extension%20for%20DuckDB)

* **CI resource usage reduction:** The project successfully reduced its GitHub Actions usage by approximately 27% following guidance from ASF Infrastructure to stay within project-wide resource limits. [Thread](https://lists.apache.org/list?dev@arrow.apache.org:lte=2M:Consumption%20of%20ASF%20Shared%20GitHub-hosted%20Runner%20Resources)

* **C Data Interface spec clarification:** Developers discussed how to handle field names containing NUL bytes in the C Data Interface, concluding that producers should error out as NUL bytes are incompatible with standard C string handling. [Thread](https://lists.apache.org/list?dev@arrow.apache.org:lte=2M:Spec%20clarification%20on%20field%20names)

* **Bug squash initiative:** Maintainers launched a call for volunteers to triage and reproduce findings for a large backlog of old issues across various language implementations. [Thread](https://lists.apache.org/list?dev@arrow.apache.org:lte=2M:Closing%20out%20old%20issues%20-%20extra%20help%20wanted)
